# DRIPLY — UX-аудит текущей версии

Дата: 23 июля 2026.

Аудит основан на текущем frontend-коде и мобильных скриншотах тестирования в Safari и Chrome. Прямой автоматический просмотр production URL в момент аудита был недоступен, поэтому выводы нужно подтвердить ручным smoke-тестом после следующего деплоя.

## Сильные стороны

### 1. Понятная основная идея

Свайп-лента сразу объясняет продукт: один товар, крупное фото, цена, размер, город и быстрые действия. Пользователь не перегружен каталогом на первом экране.

### 2. Хорошая мобильная визуальная иерархия

- Крупное изображение.
- Контрастная цена.
- Минимум текста.
- Доступные основные действия.
- Нижняя навигация соответствует привычным мобильным паттернам.

### 3. Жесты разделены правильно

Переключение фото по тапу снижает конфликт между каруселью и свайпом товара. Это правильное решение для продукта, где свайп карточки является главной механикой.

### 4. Подробная карточка уже имеет правильный каркас

Есть фотографии, цена, характеристики, описание, продавец и переход к сообщениям. Это хорошая база для будущей реальной карточки объявления.

### 5. Интерфейс уже локализован под СНГ

Русский язык, локальные города и валютная стратегия соответствуют выбранному рынку.

## Критические проблемы до подключения реальных пользователей

### P0 — данные пока не настоящие

Все ключевые экраны работают на моках. Перезагрузка страницы сбрасывает состояние. Нет реального пользователя, реальных объявлений, сохранённых свайпов и сообщений.

### P0 — отсутствует полноценная авторизация

Нужно определить веб-авторизацию до Telegram:

- email + код;
- телефон + код;
- Google/Apple как дополнительные варианты;
- анонимный просмотр с обязательным входом перед публикацией или сообщением.

### P0 — публикация товара пока не работает

Форма должна реально:

- загружать до 10 фото;
- позволять менять порядок;
- выбирать обложку;
- валидировать поля;
- сохранять черновик;
- отправлять товар в базу;
- показывать прогресс загрузки и ошибки.

### P0 — нет реального жизненного цикла сделки

Нужны действия продавца:

- редактировать;
- снять с публикации;
- зарезервировать;
- отметить проданным;
- архивировать;
- удалить.

### P1 — архитектура frontend слишком централизована

`AppStable.tsx` содержит данные, навигацию, жесты и экраны. Перед подключением API необходимо декомпозировать приложение. Иначе любое новое состояние будет повышать риск повторных багов со свайпом, фокусом input и прокруткой.

### P1 — недостаточно состояний интерфейса

Для каждого экрана нужны:

- loading;
- empty;
- error;
- retry;
- offline;
- skeleton;
- disabled;
- success.

Сейчас интерфейс в основном показывает только успешное состояние.

### P1 — доступность

Нужно проверить:

- размер touch-target не меньше 44 px;
- aria-label для всех иконок;
- контраст вторичного текста;
- фокус с клавиатуры;
- reduced motion;
- сообщения screen reader при лайке и смене фото.

### P1 — навигация браузера

Нужно перейти на React Router или аналогичную маршрутизацию, чтобы:

- кнопка назад браузера работала ожидаемо;
- у товара был отдельный URL;
- ссылкой на товар можно было поделиться;
- обновление страницы не возвращало пользователя на ленту.

## Анализ экранов

### Лента

Что хорошо:

- сильный первый экран;
- действия читаются;
- карточка адаптируется к высоте;
- фото переключаются без конфликтующих свайпов.

Что добавить:

- реальный индикатор загрузки следующего товара;
- состояние конца ленты;
- отмена последнего действия с серверным сохранением;
- подпись или onboarding для жестов только при первом запуске;
- защита от двойного срабатывания;
- preloading следующих фотографий.

### Поиск

Что добавить:

- debounce;
- недавние запросы;
- подсказки брендов и категорий;
- результаты с количеством;
- понятное пустое состояние;
- сохранённые фильтры;
- сортировка;
- очистка запроса одной кнопкой.

### Фильтры

Минимальный набор:

- страна;
- город;
- радиус или доставка;
- категория;
- бренд;
- размер;
- цена от/до;
- состояние;
- цвет;
- пол/унисекс;
- только с доставкой;
- сортировка по новизне, цене и популярности.

### Карточка товара

Добавить:

- дата публикации;
- просмотры;
- способы доставки;
- возможность пожаловаться;
- поделиться ссылкой;
- похожие товары;
- другие товары продавца;
- безопасные подсказки по сделке;
- статус резерва/продажи;
- sticky-кнопку сообщения на длинном экране.

### Профиль

Текущий обновлённый каркас правильный. Следующий уровень:

- реальные данные;
- редактирование;
- рейтинг и отзывы;
- подписка;
- табы с настоящей фильтрацией;
- меню управления объявлением;
- пустые состояния;
- публичный и собственный профиль должны отличаться.

### Сообщения

Нужны:

- список диалогов;
- экран одного диалога;
- связь диалога с конкретным товаром;
- отправка текста и изображений;
- статусы доставки/прочтения;
- блокировка и жалоба;
- быстрые ответы;
- уведомления.

## Производительность

До большого количества товаров необходимо:

- использовать responsive images;
- генерировать thumbnails;
- lazy-load сеток;
- prefetch следующего товара;
- ограничить размер исходных загрузок;
- сжимать изображения на клиенте или сервере;
- кешировать API-запросы;
- виртуализировать длинные списки.

## Безопасность и доверие

Для маркетплейса обязательны:

- подтверждение контакта;
- жалобы;
- блокировка пользователей;
- модерация объявлений;
- запрет запрещённых товаров;
- история действий;
- защита service-role Supabase только на backend;
- RLS для пользовательских таблиц;
- rate limiting;
- безопасная обработка изображений.

## Итог аудита

Визуально продукт уже выглядит как убедительный мобильный прототип. Главный риск сейчас не дизайн, а переход от интерактивного mock-интерфейса к надёжному marketplace-продукту. Следующий этап должен быть не добавлением новых декоративных экранов, а архитектурной подготовкой и подключением реальных данных по одному вертикальному сценарию: пользователь → публикация товара → лента → избранное → сообщение продавцу.
