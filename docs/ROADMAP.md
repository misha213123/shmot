# DRIPLY — план разработки

## Принцип

Делаем продукт вертикальными законченными сценариями, а не набором несвязанных экранов. Каждый этап должен завершаться работающей production-функцией.

## Этап 0 — стабилизация и архитектура

Цель: подготовить frontend к реальным данным без повторения мобильных багов.

- Разделить `AppStable.tsx` на экраны и компоненты.
- Добавить React Router.
- Вынести типы товаров, пользователей и сообщений.
- Создать API client.
- Добавить store для сессии, региона, избранного и ленты.
- Создать единые компоненты loading / empty / error.
- Настроить lint, typecheck и production build.
- Добавить smoke checklist для Safari и Chrome.

Готово, когда существующий UI работает так же, но код модульный и сборка проверяется автоматически.

## Этап 1 — пользователи и авторизация

- Supabase Auth.
- Вход по email или телефону.
- Анонимный просмотр.
- Создание профиля.
- Выбор страны, города и валюты.
- Редактирование профиля.
- Аватар в Supabase Storage.
- Публичный и собственный профиль.

Готово, когда пользователь может зарегистрироваться, войти, выйти и сохранить профиль.

## Этап 2 — реальные объявления

- Таблицы products и product_images.
- Создание товара.
- До 10 фотографий.
- Выбор главной фотографии.
- Изменение порядка.
- Компрессия и загрузка.
- Черновики.
- Редактирование.
- Удаление.
- Статусы active / reserved / sold / archived.
- Страница товара с постоянным URL.

Готово, когда продавец публикует реальный товар, а другой пользователь видит его после обновления страницы.

## Этап 3 — настоящая лента и свайпы

- API ленты.
- Фильтрация по стране и городу.
- Сохранение swipe events.
- Избранное в базе.
- Undo последнего свайпа.
- Исключение уже просмотренных товаров.
- Preload следующей карточки и фото.
- Состояние конца ленты.

Готово, когда свайпы сохраняются между устройствами и сессиями.

## Этап 4 — поиск и фильтры

- Полнотекстовый поиск.
- Бренды и категории.
- Размеры.
- Цена.
- Состояние.
- Цвет.
- Страна и город.
- Доставка.
- Сортировка.
- Недавние запросы.
- Сохранённые фильтры.

Готово, когда пользователь может найти конкретный тип товара без использования ленты.

## Этап 5 — сообщения

- Таблицы conversations и messages.
- Диалог привязан к товару.
- Список диалогов.
- Отправка текста.
- Отправка фото.
- Прочитано / не прочитано.
- Блокировка пользователя.
- Жалоба.
- Уведомления в браузере позднее.

Готово, когда покупатель и продавец могут договориться внутри DRIPLY.

## Этап 6 — доверие и сделки

- Рейтинг продавца.
- Отзывы после завершённой сделки.
- Резерв товара.
- Подтверждение продажи.
- Жалобы на товары и пользователей.
- Модерация.
- Верификация профиля.
- Правила безопасной сделки.
- Логи действий администратора.

## Этап 7 — рекомендации

### Версия 1: rules-based

Баллы товара зависят от:

- страны и города;
- подходящего размера;
- любимых категорий;
- брендов;
- ценового диапазона;
- свежести объявления;
- качества продавца;
- лайков и пропусков пользователя.

### Версия 2: поведенческая

Добавляются события:

- открытие товара;
- время просмотра;
- просмотр нескольких фото;
- добавление в избранное;
- сообщение продавцу;
- покупка;
- скрытие похожих товаров.

### Версия 3: ML ranking

После накопления достаточного количества событий можно обучить модель ранжирования. До этого сложный ML не нужен.

## Этап 8 — рост и удержание

- Подписки на продавцов.
- Уведомления о новых товарах.
- Снижение цены.
- Подборки.
- Реферальная программа.
- Boost / продвижение объявлений.
- Аналитика продавца.
- Система достижений или репутации.

## Этап 9 — Telegram

Только после стабильной веб-версии:

- Telegram WebApp SDK.
- Telegram login.
- Deep links на товары.
- Уведомления через бота.
- Открытие диалогов и объявлений из Telegram.

## Ближайший спринт

Порядок следующих работ:

1. Декомпозиция frontend без визуальных изменений.
2. Реальная авторизация и профиль.
3. Реальное создание объявления с фотографиями.
4. Реальная лента из базы.
5. Избранное и свайпы в базе.
6. Поиск и фильтры.
7. Сообщения.

Новые декоративные функции до завершения пунктов 1–5 добавляются только при необходимости исправить UX.
