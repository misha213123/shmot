# DRIPLY — принятые решения

## Продукт и рынок

- Продукт создаётся прежде всего для стран СНГ, а не для ЕС.
- Telegram Mini App пока не является приоритетом.
- Сначала строится полноценный веб-маркетплейс.
- Основная механика discovery — карточки товаров и свайпы.
- Основной фокус — одежда, обувь и fashion-аксессуары.

## Навигация

Основная нижняя навигация:

1. Лента
2. Поиск
3. Добавить объявление
4. Избранное
5. Профиль

Главная лента занимает видимую высоту телефона и не имеет вертикальной прокрутки. Длинные экраны прокручиваются внутри приложения.

## Жесты

- Свайп карточки влево: пропустить.
- Свайп карточки вправо: добавить в избранное.
- Тап по правой половине фото: следующее фото.
- Тап по левой половине фото: предыдущее фото.
- Вертикальная прокрутка работает только на длинных экранах.
- Фото не переключаются горизонтальным свайпом, чтобы жест не конфликтовал со свайпом карточки.

## Фотографии

- До 10 фотографий на объявление.
- Первое фото — обложка.
- Продавец сможет менять порядок фотографий.
- Оригинал сохраняется в Supabase Storage.
- Обложка ленты подготавливается отдельно.
- Для поиска используется квадратная миниатюра.
- Для карточки товара используется оригинальное соотношение сторон.
- Нужен редактор главной обложки: масштаб и позиция.

## Валюты и регионы

- Цена хранится в исходной валюте продавца.
- Страна объявления и валюта обязательны.
- Автоопределение страны не считается абсолютным источником истины.
- Пользователь всегда может вручную изменить страну.
- Лента сначала показывает локальные объявления.
- Конвертация цен может быть добавлена позднее как дополнительное отображение.

## Профиль

Профиль должен включать:

- аватар;
- имя и username;
- город и страну;
- верификацию;
- рейтинг и отзывы;
- число объявлений, подписчиков и подписок;
- активные объявления;
- проданные товары;
- архив;
- редактирование профиля;
- сообщения;
- управление каждым объявлением.

## Статусы объявления

Планируемые статусы:

- draft;
- active;
- reserved;
- sold;
- archived;
- hidden;
- deleted.

## Рекомендации

Рекомендации должны строиться постепенно:

1. География и доступность.
2. Категории, размеры и бренды.
3. Лайки и пропуски.
4. Открытия карточек и время просмотра.
5. Переключение фотографий.
6. Сообщения продавцам.
7. Покупки и завершённые сделки.

На раннем MVP не нужен сложный ML. Сначала используется понятная система баллов и правил, затем — персональная модель на накопленных событиях.

## Разработка

- Любые крупные решения сохраняются в `docs/`.
- Изменения отправляются в `main`.
- Vercel отвечает только за frontend.
- Render отвечает только за backend.
- При изменениях только `frontend/**` backend не должен пересобираться.
- Секреты никогда не коммитятся.
